# Capacitor native wrap (Epic 10)

Reference for the Capacitor packaging work ‚Äî wrapping the existing Vite/React build in a
thin native shell so Gravy can ship to the App Store / Play Store and (later) reach native
capabilities (push, biometrics, haptics, camera). CLAUDE.md links here; read this when
touching the native build path. The iOS side has graduated past the initial spike (`ios/` is
committed with real icon/splash customizations); it still stops short of signing and store
config (see `BACKLOG.md` Epic 10's "Path to first TestFlight build"). Android remains an
ungraduated spike.

## Why Capacitor (not React Native / PWA-only)

Decided in `BACKLOG.md` Epic 10: Capacitor wraps `dist/` with near-zero rewrite of `src/`,
versus rewriting all 40+ components in `src/components/` for React Native. The same Vite/React
app keeps shipping to GitHub Pages as a PWA; the native shell is an additional target, not a
fork.

## What the spike added

- **Dependencies** ‚Äî `@capacitor/core` (runtime), plus `@capacitor/cli`, `@capacitor/ios`,
  `@capacitor/android` (dev). Pinned to the `^8.x` line that was current at spike time.
- **`capacitor.config.ts`** (repo root) ‚Äî `appId: com.gravyapp.app`, `appName: Gravy`,
  `webDir: dist`.
- **A native build mode** ‚Äî `npm run build:native` runs `vite build --mode capacitor`, which
  forces a **root-relative base** (see below). `npm run cap:sync` chains `build:native` ‚Üí
  `cap sync`. `npm run cap:open:ios` / `cap:open:android` open the native IDEs.

## The base-path gotcha (the one thing that will bite you)

`vite.config.ts` sets `base` to `/Gravy/` for a normal `build` because GitHub Pages serves the
app from the `andrewpapin.github.io/Gravy/` sub-path. A Capacitor WebView instead serves the
bundled `dist/` from the **root** of a custom scheme (`capacitor://localhost` on iOS,
`https://localhost` on Android), so a `/Gravy/`-prefixed asset URL 404s every script and
stylesheet ‚Äî you get a blank white screen.

The fix is mode-driven: `--mode capacitor` selects `base: '/'`. Always build the native bundle
with `npm run build:native` (or `cap:sync`), never plain `npm run build`. Verified: the native
bundle emits `src="/assets/..."` while the Pages bundle still emits `src="/Gravy/assets/..."`.

## Local workflow

```bash
npx cap add android        # one-time: scaffold android/ (Gradle template, no Android Studio needed)
npm run cap:sync           # build:native + copy dist/ into both native projects
npm run cap:open:ios       # open Xcode  (needs macOS + Xcode to build/run)
npm run cap:open:android   # open Android Studio (or `cd android && ./gradlew assembleDebug`)
```

`ios/` is now committed, so **don't** run `npx cap add ios` again (it would overwrite the
committed customizations) ‚Äî just `npm run cap:sync` to refresh the synced web bundle, then
`npm run cap:open:ios`.

## Why `android/` is still gitignored (and `ios/` no longer is)

The native projects are generated by `npx cap add` and are fully regenerable from
`capacitor.config.ts` + `dist/`. They also embed a copy of the built web bundle
(`android/app/src/main/assets/public`, `ios/App/App/public`) and carry tooling-version churn
(Gradle wrapper, `.xcodeproj`), so committing them while they're still a bare scaffold would add
~3 MB of regenerable, noisy artifacts to every diff.

**`ios/` has graduated**: it now carries real customizations (native app icon + launch/splash
screen generated from `public/pwa-512x512.png` and `public/maskable-icon-512x512.png`) and is
tracked in git. `ios/.gitignore` (committed alongside it) still excludes the synced web bundle
(`App/App/public`, regenerated by `cap sync`), `Pods`, `DerivedData`, and generated config files
(`capacitor.config.json`, `config.xml`) ‚Äî those stay build output, not source. Re-run
`npm run cap:sync` after pulling to refresh the synced web assets before opening Xcode.

`android/` stays gitignored until an Android track is scheduled and its shell starts carrying
real customizations the same way.

## Service worker is disabled under `--mode capacitor`

`vite.config.ts` passes `disable: mode === 'capacitor'` to `VitePWA`, so the native bundle
emits **no** Workbox service worker ‚Äî inside a Capacitor WebView a SW is redundant (the shell
already bundles assets locally) and could cache-fight `UpdatePrompt`'s auto-reload. `disable`
(rather than dropping the plugin) keeps the `virtual:pwa-register` modules resolvable as no-ops,
so `UpdatePrompt`'s import and `useRegisterSW` still build; on native they simply never report
an update. The Pages/PWA build is unaffected and keeps its SW.
